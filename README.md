# VibeYeeter3000

An internal PaaS for AI-assisted application development by non-technical users.

VibeYeeter3000 lets anyone at the company build and deploy applications using AI coding agents (Claude, Cursor, etc.) without needing to understand Kubernetes, Terraform, or CI/CD pipelines. The platform provisions repos, manages infrastructure, runs deployments, and provides a dashboard where non-coders can see what's running, roll back, inspect migrations, and manage secrets.

Engineers vibe-code in their repos. The platform does the rest.

## What it is

- **For non-technical users**: a dashboard to see deployments, roll back, view migration history, manage secrets, and approve infrastructure changes — no terminal required
- **For AI coding agents**: a standardized repo structure, CLAUDE.md conventions, and automated pipelines so agents know exactly how to build, test, and ship
- **For the platform team**: a GitHub App + Terraform runner + Kubernetes operator that governs every app's lifecycle

## Core stack

| Layer | Technology |
|---|---|
| App language | TypeScript |
| App framework | Next.js 14+ (App Router) |
| ORM | Drizzle |
| Database | PostgreSQL (CloudNative PG) |
| Container runtime | Docker |
| Orchestration | AWS EKS (Kubernetes) |
| Infra-as-code | Terraform |
| CI/CD | GitHub Actions (generated by platform) |
| Image registry | AWS ECR |
| Secrets | AWS Secrets Manager + External Secrets Operator |
| Ingress auth | Cloudflare Access → JumpCloud SSO |
| Platform auth | JumpCloud SAML |
| GitHub integration | GitHub App (`vibeyeeter-bot`) |

## Local development

From a clean clone:

```bash
docker compose -f docker-compose.dev.yml up -d
cp apps/api/.env.example apps/api/.env.local
cp apps/web/.env.example apps/web/.env.local
pnpm install
pnpm --filter @vibeyeeter/api db:migrate
pnpm dev
```

(Or run `./scripts/dev-setup.sh` instead of the two `cp` lines — it does the
same thing and won't clobber an existing `.env.local`.)

This gets you:

- **Postgres 16** running in Docker on `localhost:5432` (`docker-compose.dev.yml`), with schema applied via `db:migrate`
- **API** on [http://localhost:3002](http://localhost:3002) — try `GET /health`
- **Web** on [http://localhost:3000](http://localhost:3000)
- **tf-runner** on [http://localhost:4001](http://localhost:4001) — try `GET /health`

> Ports 3002/4001 (rather than the more typical 3001/4000) are used to avoid
> colliding with other local services that may already be running on this
> machine.

### No GitHub App or JumpCloud required

The default `apps/api/.env.example` leaves `GITHUB_APP_ID`,
`GITHUB_APP_PRIVATE_KEY`, `GITHUB_WEBHOOK_SECRET`, and `JUMPCLOUD_SAML_CERT`
blank. The API starts fine without them — it logs a warning at boot and the
GitHub webhook / SAML routes simply won't function until those are set.

`DEV_AUTH_BYPASS=true` (set by default in `apps/api/.env.example`) makes
`requireSession` skip real session validation and attach a fake local admin
user (`{ id: "local", email: "dev@local", teams: ["dev"], isAdmin: true }`) to
every request. This lets the web UI and any authenticated API route work
without a real JumpCloud SSO flow. **Never set this outside local dev.**

### Resetting the local database

```bash
docker compose -f docker-compose.dev.yml down -v   # drops the postgres volume
docker compose -f docker-compose.dev.yml up -d
pnpm --filter @vibeyeeter/api db:migrate
```
